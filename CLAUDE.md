# StepBy_HP

このリポジトリはStepByプロジェクトのホームページです。バックエンド (`barrierfree-map`) とスマホアプリ (`StepBy`) はLinuxローカルサーバー上にあり、本リポジトリと連動して開発されます。

## 必読ドキュメント

作業を始める前に必ず以下を読んでください。

- [`docs/SERVER_ACCESS.md`](docs/SERVER_ACCESS.md) — ローカルLinuxサーバーへのSSH接続手順、各リポジトリの場所、横断作業の進め方

## フロントエンドのデフォルト作業対象

フロントエンドリポジトリ（Linuxサーバー `/home/otama/StepBy`）には `UI0`、`UI1`、`UI2` の複数バリアントが存在しますが、**特に指示がない場合は `UI0` を作業対象とする**こと。

- 「フロントエンドを直して」「アプリ側で〜」などUI指定のない依頼は `UI0` を編集・確認する
- 他のUI（`UI1`、`UI2`）を触る場合は、ユーザーから明示的に「UI1で」「UI2を」と指示があったときのみ
- 横断的に修正が必要な場合は、まず `UI0` を直してからユーザーに他UIへの反映可否を確認する

## バージョン更新の手順

コミットでバージョンを上げる際は、**コミットメッセージのバージョン番号と `scripts/site-config.js` の `version` フィールドを必ず一致させる**こと。

```javascript
// scripts/site-config.js
window.StepBySiteConfig = {
  version: "HP: v1.6.5",   // ← コミットメッセージの vX.Y.Z と同期
  ...
};
```

**手順：**
1. 変更内容に応じて新バージョン番号を決める
2. `scripts/site-config.js` の `version` を `"HP: vX.Y.Z"` 形式で更新
3. その他のソース変更とまとめて1コミットにする（site-config.js の更新だけを別コミットにしない）
4. コミットメッセージの1行目を `vX.Y.Z` で揃える

`site-config.js` のバージョン表記はサイト上の表示にも使われるため、コミット番号とずれていると利用者にとって混乱の元となる。

## コミットメッセージの書式

コミット・プッシュする際は、必ず `git log --oneline -10` で過去のコミットメッセージを確認し、その書式に合わせること。

このリポジトリの書式（複数行構成）：

```
vX.Y.Z
・<変更1>
・<変更2>
```

- 1行目：バージョン番号のみ（`vX.Y.Z`）
- 2行目以降：各変更を全角中黒（`・`）で始め、**1変更1行**で記述
- バージョン番号と変更の間で改行する
- 変更同士も改行で区切る（同じ行に並べない）
- 日本語で記述
- 英語のConventional Commits形式（`feat:`, `fix:` など）は使わない

例：

```
v1.6.0
・バックエンドのv1.27.2対応
・過去一週間の点字ブロック記録距離推移が折れ線グラフで見えるようにした
```

```
v1.6.3
・ユーザー数推移の折れ線グラフも表示させた
```

バージョン番号はユーザーに確認するか、変更内容に応じた妥当なバンプ案を提示してから決める。

`git log --oneline -10` だと過去の複数行コミットが1行にまとめて見えるが、これは表示形式の都合であり、実際のコミットメッセージは上記の複数行構成で作成すること。
